# ADR-0002: API Style, Framework, and Contract

## Status

Accepted

## Context

The React frontend needs a stable boundary to reuse backend behavior without
importing Python internals or duplicating business rules. The API technology was
previously undecided. The project needs a pragmatic API approach for a React
frontend and Python backend.

## Decision

Use FastAPI for the initial `src/api` implementation.

Use REST-style JSON endpoints under `/api/v1`.

Use the OpenAPI contract generated by FastAPI from day one.

Initial endpoint direction:

```text
POST /api/v1/registrations
GET /api/v1/registrations
GET /api/v1/registrations/{registration_id}
POST /api/v1/registrations/{registration_id}/process
POST /api/v1/registrations/{registration_id}/revalidate
GET /api/v1/registry/registrations
POST /api/v1/registry/refreshes
GET /api/v1/registry/refreshes/latest
GET /api/v1/registry/entries
```

The API layer should stay thin:

- parse and validate HTTP requests
- serialize responses
- map backend errors to HTTP errors
- call `src/core` services
- avoid owning validation, duplicate detection, checksum, or persistence rules

## Consequences

Positive:

- FastAPI fits the React plus Python stack well.
- OpenAPI gives the frontend/backend boundary an explicit contract early.
- Typed schemas improve API testability and documentation.
- REST maps naturally to the current registration and registry workflows.

Tradeoffs:

- FastAPI becomes a runtime dependency.
- The team must keep route handlers thin to avoid moving business rules out of
  `src/core`.
- GraphQL remains out of scope unless future discovery/search workflows need
  highly flexible nested querying.

## Follow-Up

- Add FastAPI to `pyproject.toml` when implementation begins.
- Add `src/api/` with an application entry point, route modules, schemas, and
  dependency wiring.
- Add API tests for request validation, response shape, and error mapping.
